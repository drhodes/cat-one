Lean 4 Exercise 1.1: Uniqueness of Inverses in a Group

Roman demonstrates that in any group, the inverse of an element is unique. Formally: $\alpha = \alpha \cdot 1 = \alpha \cdot (a \cdot \beta) = (\alpha \cdot a) \cdot \beta = 1 \cdot \beta = \beta$.

import Mathlib.Algebra.Group.Basic

theorem group_inverse_unique {G : Type} [Group G] (a α β : G)
    (h1 : α * a = 1) (h2 : a * β = 1) : α = β := by
  sorry
import Mathlib.Algebra.Group.Basic

theorem group_inverse_unique {G : Type} [Group G] (a α β : G)
    (h1 : α * a = 1) (h2 : a * β = 1) : α = β := by
  calc
    α = α * 1       := by rw [mul_one]
    _ = α * (a * β) := by rw [h2]
    _ = (α * a) * β := by rw [mul_assoc]
    _ = 1 * β       := by rw [h1]
    _ = β           := by rw [one_mul]

Section 2: Categories, Objects, and Morphisms in Lean 4

A category consists of a collection of objects and a collection of morphisms (often called arrows) that connect these objects. The most critical rule of a category is that morphisms must compose. If you have a morphism $f: X \to Y$ and a morphism $g: Y \to Z$, there must exist a composite morphism $X \to Z$.

In Lean 4's Mathlib, a category is defined as a typeclass parameterized by the type of its objects [cite: 2]. However, because Lean 4 is a dependently typed language with a strict universe hierarchy, the objects live in one universe (Type u) and the morphisms live in another universe (Type v). This distinction between `SmallCategory` (where objects and morphisms share a universe) and `LargeCategory` (where they do not) is crucial [cite: 5].

Mathlib provides specific, highly optimized notation for these concepts [cite: 5, 6]:

  • X ⟶ Y : The type of morphisms from object $X$ to object $Y$ (typed using \hom).
  • 𝟙 X : The identity morphism on object $X$ (typed using \b1).
  • f ≫ g : The composition of morphism $f$ followed by $g$ (typed using \gg).

Lean 4 Exercise 1: Exploring the Category Axioms

Let's look at how Mathlib formally defines the raw structure of a category. Lean requires us to provide proofs that our composition is associative, and that every object has an identity morphism that acts as a neutral element.

import Mathlib.CategoryTheory.Category.Basic
import Mathlib.CategoryTheory.Types

open CategoryTheory

-- Mathlib defines CategoryStruct roughly like this:
-- class CategoryStruct (obj : Type u) where
--   Hom : obj → obj → Type v
--   id : (X : obj) → Hom X X
--   comp : {X Y Z : obj} → Hom X Y → Hom Y Z → Hom X Z

-- And the axioms are layered on top:
--   id_comp : ∀ {X Y} (f : Hom X Y), comp (id X) f = f
--   comp_id : ∀ {X Y} (f : Hom X Y), comp f (id Y) = f
--   assoc : ∀ {W X Y Z} (f : Hom W X) (g : Hom X Y) (h : Hom Y Z), 
--           comp (comp f g) h = comp f (comp g h)

-- Let's prove a simple lemma: Composing an identity morphism with another identity 
-- morphism yields the identity morphism.
theorem id_comp_id_is_id {C : Type u} [Category C] (X : C) : 
  (𝟙 X) ≫ (𝟙 X) = 𝟙 X := by
  -- The axiom `Category.id_comp` states that `𝟙 X ≫ f = f`.
  -- Here, our `f` is just `𝟙 X`.
  exact Category.id_comp (𝟙 X)

-- Try this one yourself: Prove it using the right-identity axiom instead.
theorem id_comp_id_is_id_alt {C : Type u} [Category C] (X : C) : 
  (𝟙 X) ≫ (𝟙 X) = 𝟙 X := by
  exact Category.comp_id (𝟙 X)

Lean 4 Exercise 2: Proving Associativity in Types

Let's verify that ordinary function composition behaves precisely as categorical composition when we are operating in the category of Types. In Lean, we can define the category of Types explicitly by specifying that Hom X Y is just X → Y [cite: 8].

import Mathlib.CategoryTheory.Types

open CategoryTheory

-- In the category of Types, `X ⟶ Y` is definitionally `X → Y`.
-- And `f ≫ g` is defined as `g ∘ f` (g after f).

-- Let's prove that function composition is associative.
theorem types_comp_assoc {W X Y Z : Type} (f : W ⟶ X) (g : X ⟶ Y) (h : Y ⟶ Z) : 
  (f ≫ g) ≫ h = f ≫ (g ≫ h) := by
  -- Because `f ≫ g` is defined as `fun x => g (f x)`, Lean's definitional 
  -- equality can see that both sides of this equation are identical.
  rfl

-- Proof of identity.
theorem types_id_comp {X Y : Type} (f : X ⟶ Y) : 
  (𝟙 X) ≫ f = f := by
  -- `𝟙 X` is `fun x => x`. Thus `f (id x)` is just `f x`.
  rfl

Section 4: Epimorphisms and Monomorphisms

In set theory, we talk about injective (one-to-one) and surjective (onto) functions. In category theory, because we cannot look "inside" the objects to see their elements, we must define these properties entirely by how the morphisms compose with other morphisms. We call these Monomorphisms (monos) and Epimorphisms (epis).

Lean 4 Exercise 3: Defining Epimorphisms

Mathlib provides the Epi and Mono typeclasses. Let's look at how an Epimorphism is formally stated and prove a simple property about it.

import Mathlib.CategoryTheory.EpiMono

open CategoryTheory

variable {C : Type u} [Category C] {X Y : C}

-- Mathlib defines `Epi f` roughly as:
-- class Epi (f : X ⟶ Y) : Prop where
--   left_cancellation : ∀ {Z : C} (g h : Y ⟶ Z), f ≫ g = f ≫ h → g = h

-- If f is an isomorphism, it is automatically an epimorphism. 
-- Let's prove a basic cancellation property assuming `Epi f`.
theorem cancel_epi (f : X ⟶ Y) [Epi f] {Z : C} (g h : Y ⟶ Z) (H : f ≫ g = f ≫ h) : 
  g = h := by
  exact Epi.left_cancellation g h H

Section 5: Universal Properties (Products & Coproducts)

In category theory, we define objects not by their internal data, but by their universal properties—how they relate holistically to the rest of the category. A Categorical Product of two objects $A$ and $B$ is an object $A \times B$ equipped with two projections ($p_1, p_2$) such that for any other object $C$ with projections to $A$ and $B$, there is a unique morphism from $C$ to $A \times B$ that makes the diagram commute.

Lean 4 Exercise 4: Constructing the Product Morphism

Let's construct the unique morphism into the product object in the category of Types.

import Mathlib.CategoryTheory.Types

open CategoryTheory

-- Given an object C and morphisms f : C ⟶ A and g : C ⟶ B,
-- we construct the unique morphism into the product A × B.

def type_product_factor {A B C : Type} (f : C ⟶ A) (g : C ⟶ B) : C ⟶ (A × B) :=
  fun c => (f c, g c)

-- Now let's prove that this factorization commutes with the first projection.
theorem type_product_commutes_fst {A B C : Type} (f : C ⟶ A) (g : C ⟶ B) :
  (type_product_factor f g) ≫ Prod.fst = f := by
  -- By definition, `Prod.fst (f c, g c)` is `f c`.
  rfl

Section 6: Functors (Contexts & Containers)

A functor $F: \mathcal{C} \to \mathcal{D}$ maps every object $X$ in $\mathcal{C}$ to an object $F(X)$ in $\mathcal{D}$, and it maps every morphism $f: X \to Y$ to a morphism $F(f): F(X) \to F(Y)$. In functional programming, functors represent computational contexts or containers (like a List or an Option type).

Lean 4 Exercise 5: The Functor Laws

Mathlib defines functors rigidly. They cannot just map data; they must rigorously preserve the structure of the category [cite: 6].

import Mathlib.CategoryTheory.Functor.Basic

open CategoryTheory
open CategoryTheory.Functor

variable {C D : Type u} [Category C] [Category D] (F : C ⥤ D)

-- EXERCISE: Prove that mapping a composed morphism is the same as 
-- composing the mapped morphisms.
theorem functor_preserves_composition {X Y Z : C} (f : X ⟶ Y) (g : Y ⟶ Z) : 
  F.map (f ≫ g) = (F.map f) ≫ (F.map g) := by
  -- This is the second fundamental law of Functors. 
  -- We apply the built-in Mathlib axiom `map_comp`.
  exact F.map_comp f g

Section 7: Natural Transformations

If functors are mappings between categories, mappings between functors are called Natural Transformations. A natural transformation provides a way to systematically translate one "container" into another (e.g., safely converting a List A to an Option A) without altering the underlying data.

import Mathlib.CategoryTheory.NatTrans

Section 8: The Yoneda Lemma

The Yoneda Lemma asserts that an object $A$ is entirely and uniquely determined by the functors representing the network of all morphisms pointing into it or out of it. Philosophically, it proves that understanding how a system interacts with all other systems in its environment provides total knowledge of the system itself.
